01 / Core Design

三层正交权限平面

每一层独立控制一个维度，互不干扰，组合起来实现精细化的访问控制。

Layer 1

API / 路由权限

控制角色能否调用此端点 — 通过 Casbin 4-tuple (sub, dom, obj, act) 模型执行
middleware/require_perm.go → RequirePerm("tenant:user:view")
Layer 2

行级数据范围

控制角色能看到哪些行 — 通过 BuildDataScope 自动注入 SQL WHERE 条件
service/data_scope.go → BuildDataScope(c, "users") → WHERE agent_id = ?
Layer 3

字段级权限

控制每行中哪些字段可见 — 后端反射置零 + 前端列过滤，白名单模式（默认拒绝）
service/field_sanitizer.go → SanitizeResponse(c, "users", items)
02 / Request Lifecycle

请求处理完整链路

从 HTTP 请求到达到清洗后的 JSON 响应返回，每个环节自动执行。

1

UserAuth() 中间件

验证 session / access token，写入 context（id, role, role_id, agent_id）。执行租户边界检查。

middleware/auth.go → authHelper()
2

RequirePerm() 中间件

提取 userId + agentId，调用 Casbin Enforce。
sub="user:123" dom="5" obj="tenant:user" act="view"

middleware/require_perm.go → service.HasPermission()
3

WithResource() 中间件

将 ResourceConfig 注入 gin.Context，启用自动数据范围限定和字段清洗。

middleware/resource.go → WithResource("users")
4

ScopedList[T]() 查询

自动执行 BuildDataScope → ApplyScope（注入 WHERE） → GORM 分页查询。

service/scoped_query.go → ScopedList[model.User](c)
5

SanitizeResponse() 清洗

查询 role_field_permissions（30s 缓存），通过反射遍历 struct，denied 字段置零。

service/field_sanitizer.go → SanitizeResponse()
6

返回清洗后的 JSON

被拒绝的字段已被置零（int→0, string→"", time→zero）。前端配合 useFieldPermColumns 过滤列。

common.ApiSuccess(c, pageInfo)

04 / Role Hierarchy

角色层级与继承

两个层级概念严格分离 — parent_id 组织树与 Casbin 用户-角色映射互不关联。

🌳

parent_id 组织树

仅用于管理权限判断（谁能编辑 / 删除谁）。通过 BFS 遍历 + 5 分钟内存缓存。不参与 Casbin 权限继承。

🔗

Casbin g-policy

user → role 直接映射，没有 role → role 继承。EffectivePermCodes == DirectPermCodes（显式禁用角色间继承）。

🔀

多角色合并策略

字段权限：并集（任一角色允许 = 允许）
数据范围：取最宽
platform:* 跳过所有限制

数据范围类型

🌍
all
看到所有数据
无 WHERE 限制
🏢
agent
仅本租户数据
WHERE agent_id = ?
👥
self_and_sub
自己及下属数据
WHERE user_id IN (...)
🎯
custom
自定义范围条件
scope_val 指定
👤
self
仅自己的数据
WHERE user_id = ?

权限 Code 派生规则

// 从 FeatureSeed 的 actions 自动派生 permission code

"menu"   action  →  "menu:<featureKey>"          // e.g. "menu:channel"
"read"   action  →  "<namespace>:<featureKey>:view"  // e.g. "tenant:log:view"
"create" action  →  "<namespace>:<featureKey>:create"
"update" action  →  "<namespace>:<featureKey>:update"
"delete" action  →  "<namespace>:<featureKey>:delete"

// 超级通配符 — Casbin matcher 特殊处理
"platform:*"  →  匹配所有 obj + act，跳过字段和数据范围限制

06 / Bootstrap

种子数据引导

启动时通过种子数据自动初始化权限系统，确保默认角色和权限就位。

创建功能目录

从 defaultFeatures 种子数据创建 permission_features 和 permission_modules 记录。

model/permission_seed.go → EnsureDefaultFeatures()

派生权限 Code

根据每个 Feature 的 actions 自动派生 permission code（menu:xx, tenant:xx:view 等）。

model/permission_seed.go → EnsureDefaultPermissions()

绑定角色权限

通过 rolePermMapping 为每个系统角色分配权限集合，写入 role_permissions 和 casbin_rule。

model/permission_seed.go → rolePermMapping → CasbinSetRolePermissions()

设置数据范围

通过 defaultDataScopes 为系统角色配置每个资源的行级数据范围。

model/rbac_scope_seed.go → defaultDataScopes

设置字段白名单

通过 defaultFieldPermissions 为系统角色配置每个资源的字段可见性（白名单模式）。

model/rbac_scope_seed.go → EnsureDefaultFieldPermissions()

Casbin 集群同步

Redis SETNX 分布式锁确保只有一个 Pod 执行 casbin_rule 重建。其余 Pod 等待后 reload policy。

service/casbin_sync.go → EnsureCasbinSync()

✅

正确用法

// 组件顶层调用 Hook
const visibleColumns = useFieldPermColumns(
  columns, 'users'
);

// early return 放在 Hook 后面
if (loading) return <Spin />;

return <Table columns={visibleColumns} />;
❌

错误用法（会白屏）

// 内联 JSX 调用 — Hook 数量不一致 → 白屏
<Table columns={
  useFieldPermColumns(columns, 'users')
} />

// 条件分支中调用 — 同样违反 Hook 规则
if (visible) {
  const cols = useFieldPermColumns(...);
}
